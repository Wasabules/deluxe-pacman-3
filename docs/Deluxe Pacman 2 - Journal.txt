@2:07AM (0.2 build 239)
   I spent MANY hours working on how to set up the display.  In fullscreen mode there was a bug (with the library, it will be fixed for the next stable release, v5.0.8 I assume) that caused it to lag a little.  I spent a lot of time trying to figure out how to get around this.  There was one solution but I felt that it could be alot more work than it was worth.  ALLEGRO_FULLSCREEN_WINDOW, it's a mode where you get a fullscreen mode, BUT, in reality it's really just a window the size of your screen with the border turned off!  There are a lot of benefits to this, one being that you can ALT+TAB between the game and other programs MUCH faster, the game runs smoother in a window (who would have thought the day would come eh?).  The ONLY downside is that it sets the window size to whatever the desktop resolution is of the person playing the game (which is why it is faster, it is using the native resolution).  Well, after much searching and reading and experimenting I found Allegro TRANSFORMS!  What this does is basically moves and scales anything I draw by a certain amount (and 3D acceleration applies to this!).  So I set it to move and scale my screens to fit the screen the player has, and then create black boxes on either side of the screen so it looks identical to setting a fullscreen mode with the 800x600 screen in the center, only this is much faster and you don't have to worry about certain monitor types screwing up the screen on the player.  I'm kind of glad that there was a bug or I may have never bothered trying this idea out!!  It's funny how that seems to work out.  I'm VERY pleased, and at 2am+, tired and time for a break+sleep. :)  Now THIS is the kind of programming I used to do many years ago, up until 2am tapping away like mad on the keyboard, awesome!!! :D

   Okay, after much thought, I decided to switch back to programming in C++ rather than C99 for this project. @4:20PM (0.3.398.2211)
   I done some more work on the graphics.  I have a new red ghost, new blue (scared) ghost.  I have been using trueSpace 6.6 and Photoshop to create them.  I had problems figuring out how to do the wavey bottoms of the ghosts as it was too difficult to render such a thing the way I wanted to do it.  I ended up using the old templates I used with tile studio to create the bottoms for the old version of the game.  They are basically masks that go at the bottom of the ghost image. They're coloured a bright pink colour so I can then easily erase  them.  Once they are pasted over top of the bottom of the ghost, and then erased, what is left is the wavey pattern at the bottom of the ghost I like, works great.  I saved all images using Photoshop to preserve the alpha channels and for the first time  the game will have nice graphics that smoothly blend with the game around the edges instead of jagged edges.  I use PNG images to store them as well, which can store alpha channels and it is a lossless image format with decent compression.
   I just got done working on the new map format as well, there's quite a bit more to it over the old version as it will store the size of the map so it will be flexible for future expansion without having to convert your maps.  It stores the starting location of the player and ghosts so they can be changed.  It stores whether or not a pill of any size, is a powerpill, this means you can make any pill a powerpill, which will lead to some interesting levels where you could make the player figure out which pills are the power pills. @3:27AM (Pace 2)
	Okay, I confirmed my suspicions about what was causing the lines to appear on the screen in full screen mode using linear filtering to improve the look.  The problem lies with using sprite sheets as I do for my game.  When you use linear filtering it samples all surrounding pixels to help smooth things out.  When you have a sprite sheet, with all graphics on one bitmap, you can probably guess what happens.  Around the edges it samples pixels from the neighbouring sprites on the same bitmap.  So either I A) create new sprite sheets with a 1 pixel border around each tile that is basically a duplicate of the border pixels, which to me is not a good option.  or B) separate the tiles.  What I will do is when I load the sprite sheets, I will separate them in the game itself and copy them into separate bitmap arrays, then just destroy the original.  This way I can still store tham as sheets, no extra memory wasting border.  Also, a 1 pixel border sounds okay, but what if I want to change filtering methods in  the future that samples more than 1 pixel?  Also, the 1 pixel border idea doesn't work for MIP MAPPED or Anisotropic filtering, not that it's an issue with this 2 game, but for 3D games it would be, so I would rather stick to methods that benifit both styles of games, just to keep my coding style flexible.

@12:43PM (0.7.703.3818)
	Okay, I changed the "pixel perfect" collision detection to just normal bounding box collision detection as pixel perfect is simply too slow to be practical.  It's also not really nessecary for a game like this.  The problem with the collision detection I used in the old game was that I didn't use bounding box at all with it, I checked which tile the characters occupied and if they were on the same one, collision, the problem with this is that pacman could be just leaving a tile at the same time a ghost was just entering it, and there wouldn't be a collision, but it would detect it as one.  With bounding box it will be far more accurate, the space their bitmap images occupy will have to overlap in order to detect a collision.  There is still room for error as the pitmaps are square and have empty space around the actual images that could potentially overlap and be falsely detected as a collision.  My solution for this will probably be to reduce the size of the bounding box to be a few pixels smaller than the actual bitmap size so they can overlap a LITTLE BIT without a collision detection, say around 5 pixels or so, I'm about to check the bitmaps now to determine a good number to allow for overlap.  It should make people happy to see a ghost actually touch them... barely, yet get away without dieing rather than be like before when they got killed without a ghost touching them.  This new system should be much better, just as fast to detect without bogging the game down like pixel perfect would.  The reason why pixel perfect slows down so much is that I have to copy the bitmaps to a new temporary bitmap due to the fact that the pacman sprite is animated and can be rotated and so it needs to be rotated to it's current position the same way it is drawn on the screen to have an accurate collision test, this takes time, then each pixel in the overlapping areas needs to be checked to see if any overlap... far too slow a process for such a small benefit, it's just not worth doing.

   I added in the code needed to start displaying pickups on screen.  At first my graphics were exactly 50x50 pixels in size and each food image went right to the edge of the graphic which causes problems because I have them all one one large "sprite sheet" and they "touch" each other.  Now with older versions of Allegro this was not a problem, but due to the fact that Allegro 5 is hardware accelerated, some of the blending can cause neighbouring graphics to "bleed" over onto the image being displayed, so I had to make sure there was a thin transparent border around each image to prevent this.  Plus 50x50 was a pixel or two too large for some maps and reducing them a tad would look a little nicer.  To do this I loaded them into photoshop one at a time (I really need to learn to do scripting for these tasks!), and I first reduced the image size from 50x50 to 48x48, then I increased the canvas size back up to 50x50.  The canvas size doesn't effect the image itself, just the overall size of the image, that is to say, if you make it larger, it adds empty space around the border, anchoring the main image to the center or sides as you prefer, so this way I basically added a transparent border around each image and reduced the main image size just enough so they fit and prevent bleeding over to neighbouring graphics so I can continue to use sprite sheets.  Once these sprite sheets are loaded into the game I then have an array of images, one for each food pickup and I create a sub-bitmap for each one, it's basically like a "pointer" which points to an area on a main image and you can use it like a separate image.  This way I only have to load ONE image rather than 35, it also saves on disk space etc, it's just good practice I think.

@1:26PM v13.08.01 (1326)
   I changed the collision detection from bounding box collision, to circular collision.  The difference is that in bounding box collision you first place a rectangular box around two object you wish to test for collisions, you then check to see if these boxes overlap, if they do than the objects have collided.  For most things this is fine, but if you have characters in your game that are more round, or circular as mine are, than you will get false collisions if they overlap diagnolly as there are no corners on them and so diagonal overlaps will most likely just be empty space.  The best solution for this is to use a circular collision, which is just as simple to implement as bounding box, perhaps even easier in fact.  You simply take a radius which you are comfortable with on each object and check the distance between the two objects.  If the distance between the two objects is less than the sum of their two radiuses than you have a collision.  The math to do this is very simple.  First you subtract the distance of object1's x from object2's x and get the distance of x, then you do the same for y.  Then you get the absolute value of the square root of (distx*distx)+(disty*disty) and if that is less than or equal to the sum of the two radiuses than you have a collision.  It's easier than it sounds and there is not much code involved at all, it's very fast in fact. Trust me, I am not that good in math so if I can understand it, you can! ;)
   Anyhow, I am quite pleased with this.  I didn't like the diagonal collision problems I was seeing, and this has worked out perfectly.  Bad collisions were one of the main complaints of the first game which has even worse collision detection that bounding box.

   Okay, while testing the "WEBSITE" button in real fullscreen mode (ALLEGRO_FULLSCREEN), when you clicked on the game's icon to return to the game you were greeted with a black screen.  There is a problem with Direct3D where the game can "lose the display" and the game needs to detect when it loses the display and handle it.  Well, Allegro has some special flags you can set which tell it to preserve textures (the screen basically) so you don't lose it when the game loses the display.  You also have to detect when the game "finds" the display again and then redraw the screen so the player sees it.  Well, since I never encountered this problem before, I had set up the game so it didn't preserve the texture (this can improve performance slightly).  The problem was, you got a black screen when you got back.  Also, I wasn't testing to see if the display was lost.  So I fixed all that and it works nicely in fullscreen mode.  Interestingly enough, you don't lose display when you use the game in Windowed or what is called Fullscreen Windowed (ALLEGRO_FULLSCREEN_WINDOWED) which is basically a fullscreen, borderless window that LOOKS like fullscreen, but is in fact a window.  The benefit of this is you can switch very quickly between your game and your web browser or whatever, and even have other programs overlapping the game, I prefer it these days, even though true fullscreen mode is faster, on modern hardware I don't notice the difference. The problem is, this takes up a LOT of RAM, especially Video RAM which is normally not a problem, but on a low end system with a built in video card, it is a problem because they don't have much to start with.  Video RAM is separate from normal system memory, older cards have very little because, quite frankly, if you're not doing a lot of hardware accelerated 3D graphics, you only need enough to display your screen, which in my wife's case is 1280x1024x32bit, which equals 41,943,040bits or 5.2Kilobytes, not a lot, until you add in a buffer (you draw your scene on a separate area called a buffer which you do not see, once it is drawn, the whole thing is displayed at once, and while you're looking at that frame, another one is drawn in the background, in the buffer, so you need memory for the main display you see, plus an equal amount for the buffer).  Then there's all the graphics, which CAN be loaded into system memory, but that is AWEFUL slow for displaying so I load it all into video memory.  Well anyhow, I redone the code so that I created the buffer, then loaded in the temporary graphic, copied it to the buffer right away (rather than later) so I could delete the memory it was using.  I done this with another graphic as well, created the temporary buffer, loaded the graphic, copied it to the buffer then deleted the original right away, this way not a lot of memory was being used at any one time.  This ended the crash on her system (remind me to put a better video card in her system! heh).  I still had the horrible slowdown.  I was about to lose my mind with frustration when I finally figured out WHY the prerelease was so much faster than this!  The solution was that when I went to prepare the prerelease, I set the vsync so it was hard coded at OFF, you can't change that setting on it (try, I dare you! ;)).  But in this version you can.  Well, when I hard coded it, I put the number in to keep it off, rather than reading in a variable, so it worked fine.  But in this version it reads in a variable according to your setting.  That variable was set as a boolean type, which is either TRUE or FALSE, nothing else, because originally I had wanted to be able to toggle vsync ON or OFF.  Well, then I discovered that you can set VSYNC to 3 settings, the driver default (so you use display settings on windows for it), or forced on, or forced off, so I changed it to accommodate that, 0, 1 or 2... only problem is, I FORGOT TO CHANGE THE VARIABLE type from a boolean (bool) to an int!  Bool is true (1) or false (0).  So when it gets a 2, it probably interprets that as a 1 because it is a positive value and sets it to 1, which is ON rather than 2 which is OFF!  Argh. @3:43pm
   I watched some videos on Youtube on using Audacity, which is the free software I use to edit my sound effects, and learned a bit more about using it.  I got the idea to examine sound effects I have hear in other games to get some idea how they were edited as some of them sounded really nice and I noticed they had great stereo separation.  Well, after examining a few I noticed a trick that was used which really improved the feeling of stereo separation.  It's hard to put into words.  I also wanted to edit a few effects, most of which were all MONO sounds, no stereo (except the music), so  that the sound appears to move between one speaker to the other and back.  Some rapidly back and forth, some gradual, depending on the effect.  The powerpill effect rapidly goes back and forth between the left and right speakers to good effect.  What I did for it was I took the mono sample, then duplicated it and created a left and right stereo track.  I then alternated the volume levels so one side was louder than the other and so would give the desired effect.  For other samples, where I wanted just a greater... stereo separation, I found that duplicating the mono track into a left and right stereo pair.  Then taking the right track and offsetting it just a little created a really good effect.  You can't really hear how it is offset as I didn't move it too much, but what I think happens is that our brains are designed by God in such a way so that it senses the difference in sound from right and left and when there is a minor difference, it causes us to perceive it differently.  In any case, it worked well.
   After some testing, I found some sound effects were better left as MONO... and that is things like PILLS, and Ghost sounds when you eat them, they say "ouch" etc... those sounds come from a specific area of the screen and it is important that the player hears the sound as if it is "coming" out of a specific spot on the screen.  In the game, when you eat a pill, the game calculates roughly where on screen from left to right the sound comes from and adjusts the volume on the right and left speakers so it sounds like it is coming from that general area of the screen.  This is more difficult if the sound effect is stereo to start with, so I left them mono.  I adjusted some of the volume levels as some seemed kind of low.  After testing the game out again, it seems much better.  Some sound effects, like eating a powerpill have stereo effects as they effect the entire screen, and with the teleport, you travel from one point to another, so I have a nice stereo effect that goes from one speaker to the other suddenly, which sort of reflects the idea of transporting from one point to another.

@2:30am v17.04.24 (0228)
   I redone all the audio and music files.  I originally saved them with the maximum quality setting OGG Vorbis, which is 10.  I really didn't understand the quality differences at the time.  I later realized you don't need anywhere NEAR that.  On average, a setting of 5 will due.  For my game, most of my sound effects were done with much lower quality (none higher is needed for what they are) so I was able to reduce the quality setting down to 2, which still ended up storing the sound effects at a slightly better quality than they were at originally (88KHz for most up to 90Khz roughly, it varies).  There is no noticable quality difference at all in how they sound but WOW are they smaller files!!!  The music files dropped from over 4 megs down to under 1!  All the sound effects were similar.  In the end, the .PAK file dropped from around 12Megs down to 6Megs, half the size.  I am also thinking that this could possibly increase the smooth running of the game if it doesn't have to play at a higher quality setting.  Plus the sound files will use far less memory which could help some people.
   I am playing with the idea of a Deluxe Pacman - Remastered edition and the music files, which are OGGs on this game are the largest files.  Well, on my new project, which now uses SDL2 supports the original Amiga MOD music files which is what the music was originally, so that is even greater space savings as the MOD files are one tenth the size of the smallest OGGs with no quality reduction at all!  If only I could do that with this game and Allegro, but I am in no mood to fiddle with this library any longer.  It was great while it lasted.
